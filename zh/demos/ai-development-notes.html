AI 辅助开发笔记

Prototype April 28, 2026

概述

使用 AI 编码助手的结构化方法，在加速开发速度的同时保持代码质量、安全性和工程自主权。

挑战

AI 编码工具（GitHub Copilot、Claude、Cursor 等）可以极快地生成代码，但也引入了新的风险：

  • 你不完全理解的代码
  • 来自训练数据的安全漏洞
  • 与代码库不一致的模式
  • 过度依赖降低团队技能

此演示展示如何负责任地利用 AI 的能力。

框架：AIR 模式

Assert（断言）→ Implement（实现）→ Review（审查）

阶段1: Assert（定义问题）

在请求 AI 编码之前，明确地定义：

# 任务: 用户认证模块

## 需求
- 邮箱/密码登录
- JWT token 生成（24小时有效期）
- 使用 bcrypt 进行密码哈希
- 速率限制（每分钟5次尝试）
- 退出登录时使会话失效

## 约束
- 必须使用现有 PostgreSQL 数据库
- 遵循项目的错误处理模式
- 除 bcrypt 外不引入外部依赖
- 启用 TypeScript 严格模式

## 成功标准
- 所有单元测试通过
- 安全审计通过 OWASP 检查清单
- 性能: 认证时间 <100ms
- 代码覆盖率 >85%

## 上下文
- 现有用户模型: models/user.ts
- 数据库连接: db/connection.ts
- 错误处理工具: utils/errors.ts

阶段2: Implement（在指导下生成）

为 AI 提供具体的上下文和示例：

  • 展示现有代码模式（错误处理、异步模式等）
  • 明确技术栈和约束
  • 要求包含 JSDoc 注释
  • 要求输入验证

阶段3: Review（人工验证）

系统化审查清单：

## 代码审查清单

### 安全 ✓
- [ ] 密码哈希使用 bcrypt，salt 轮数 ≥10
- [ ] 无硬编码的密钥或 API key
- [ ] SQL 注入防护（参数化查询）
- [ ] 已实现速率限制
- [ ] Token 过期配置正确
- [ ] 敏感数据未记录到日志

### 质量 ✓
- [ ] 代码遵循项目风格指南
- [ ] 错误消息对用户友好
- [ ] 生产代码中无 console.log
- [ ] 正确使用 async/await
- [ ] 类型定义完整
- [ ] 边界情况已处理

### 测试 ✓
- [ ] 所有公共函数有单元测试
- [ ] 测试覆盖率 >85%
- [ ] 边界情况已测试（无效输入、过期 token）
- [ ] 认证流程的集成测试
- [ ] 安全测试（暴力破解、token 篡改）

### 文档 ✓
- [ ] 公共 API 有 JSDoc 注释
- [ ] README 中有使用示例
- [ ] 安全注意事项已记录
- [ ] 错误代码已记录

演示组件

1. 提示模板管理器

组织可复用的提示模板，定义角色、上下文、任务、需求、约束和输出格式。

2. 代码审查助手

自动化的初步审查：检查硬编码密钥、SQL注入风险、密码哈希、console.log 使用、any 类型等。

3. 实现跟踪器

跟踪 AI 生成 vs 人工编写的比例，生成审计报告。

工作流演示

分步示例

任务：创建密码重置功能

1. Assert 阶段（5分钟）

定义用户故事、技术需求（生成安全 token、存储与过期、发送邮件、验证 token、更新密码）、安全考量（加密安全、速率限制、不泄露邮箱是否存在、重置后强制退出所有会话）。

2. Implement 阶段（AI生成，2分钟）

AI 生成完整的 TypeScript 实现，包括请求重置和重置密码的函数。

3. Review 阶段（人工，10分钟）

人工改进：

  • ✅ 添加全面的日志记录
  • ✅ 使用 try-catch 改进错误处理
  • ✅ 提取邮件发送为独立函数
  • ✅ 添加密码强度验证
  • ✅ 使用事务保证数据一致性
  • ✅ 添加 JSDoc 文档
  • ✅ 包含安全元数据（IP、User-Agent 占位符）
  • ✅ 更好的邮件 HTML 格式
  • ✅ 重置后发送确认邮件
  • ✅ 标准化邮箱输入

跟踪的指标

指标 目标 实际
vs 手动编码节省时间 50% 65%
代码审查时间 <15分钟 10分钟
AI 引入的 Bug 0 0
捕获的安全问题 100% 100%
开发者满意度 >4/5 4.5/5

最佳实践总结

  1. 始终审查 AI 代码 - 不理解每一行就不提交
  2. 保持测试覆盖率 - AI 不会自动编写全面的测试
  3. 记录 AI 的角色 - 对 AI 生成的内容保持透明
  4. 架构决策由人来做 - AI 实现，人来设计
  5. 跟踪指标 - 衡量 AI 是否真正提升了效率和质量

结论

AI 辅助开发不是要取代开发人员——而是在保持控制、质量和安全的同时放大他们的能力。AIR 模式（Assert → Implement → Review）提供了一种结构化的方法，在不牺牲标准的前提下提升速度。

把 AI 当作需要仔细监督的高级初级开发人员，而不是自主编码的神谕。
