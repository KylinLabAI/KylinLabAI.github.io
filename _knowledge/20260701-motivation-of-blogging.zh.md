# 独立开发者一个月复盘：为什么我决定从今天开始写博客

从下定决心做独立开发者到今天，刚好满一个月。

这一个月几乎每天从早忙到晚，有时候晚上十二点还在改代码。但如果此刻让我坐下来，不带任何记录地盘点一下「这一个月到底做了什么」，脑子里最先冒出来的只有两个字：**很忙**。

具体做了什么？解决了什么问题？哪些决策是对的、哪些是弯路？

答不上来。或者说，答得很模糊。

时间过得太快了。如果不记录，这种「忙碌但模糊」的状态会一遍又一遍地重复——下个月的今天，我可能还是同样的感受：忙了一整月，却说不清忙了什么。

所以我决定开这个博客，从今天开始，把每天做的事、遇到的问题、产生的想法都写下来。既是给自己留一份可追溯的工作记录，也是提前对外分享独立软件的开发进度和思考。

> 这篇文章既是第一篇，也是一个承诺：从今天起，不做「凭感觉做事」的独立开发者。

---

## 一个月到底做了什么：列出来比我以为的多

虽然凭脑子想不清楚，但真把做过的事项一条条列出来，其实也搭了不少东西。分五类来看：

### 基础设施：从 0 到有，先把地基打好

独立开发和在公司做项目最大的不同，就是什么基础设施都得自己来。以前公司有运维帮你管服务器、有行政帮你弄域名备案、有平台团队帮你弄 CI/CD。现在这些全是自己的活。

第一个月在基础设施上的主要产出：

- **阿里云服务器采购与部署**：选定配置、初始化系统、安全组与防火墙规则配置
- **域名购买、网站开发与上线**：买下 kylinlab.tech 域名，写官网、部署静态站点、上线运行
- **SSL 证书与 DNS 配置**：全站 HTTPS、子域名划分、DNS 解析验证
- **工信部 ICP 备案和公安联网备案**：两套备案流程完整走通，这在国内做公开网站是绕不开的门槛
- **GitHub Pages / Gitee Pages 静态站点配置**：为后续的文档、博客、演示站点准备好两套国内加海外的静态托管方案

### AI 工作流：选型完成，主力工作流稳定跑起来

这一个月花了相当多的时间在 AI 工具链上——毕竟做独立开发，AI 是最核心的效率杠杆。

完成的工作：

- **横向对比了多个 AI Agent 平台和模型**（这部分我单独写过一篇详细的横测文章，感兴趣的可以翻一翻）
- **最终确定主力工作流：ClaudeCode 加 DeepSeek-V4-Pro**。前者作为 VS Code 里的 AI 编码助手，上下文和代码理解能力拉满；后者作为按量付费买 Token 的推理模型，成本可控、响应速度快，日常编码与文档场景都能覆盖

这套组合现在跑了快两周，体验很稳定。一个核心原则：**用通用客户端加按量买 Token，而不是给每个独立 AI 产品单独订阅。** 省下来的钱和切换成本都很可观。

### 产品与工具开发：两款工具已经成型

这个月最大的实体产出，是两款已经能实际用起来的工具产品：

1. **「豆匣 DouXia」——AI Chat 管理工具**：用来集中管理多平台 AI 对话、历史记录、Prompt 模板。作为 AI 工作流的日常入口，现在每天打开的频率比浏览器还高。
2. **「录匣 Recxia」——培训录制工具**：面向企业培训与技术分享场景的录屏加课件管理工具，核心功能已经开发完成，后续会加上自动打时间戳和 AI 字幕。

除了两款产品本身，还完成了：

- **微信小程序、微信公众号申请与配置**：为两款产品的国内分发与用户触达做好渠道准备
- **搭建 GitHub Actions 自动化发布流水线**：代码 push 后自动构建、打包、发布，不再需要手动在三台机器上各跑一遍
- **设计并实现用户反馈收集系统方案**：产品上线后最缺的就是真实反馈，提前把反馈入口、工单流转、问题追踪链路设计好

### 项目管理：从「想到哪做到哪」到有工具链

做独立开发最容易踩的坑，就是想到哪做到哪。我花了一些时间专门选型并搭建了一套跨平台的项目与时间管理工具链（这部分也单独写过文章分享）。核心目的只有一个：**让我每天早上打开电脑，30 秒内知道今天必须交付什么。**

### 技术学习：往深里补基础

产品开发之外，也专门留时间往深里啃了两类技术：

- **深入理解 LLM 技术细节**：不满足于调 API，而是把 Tokenization、Attention 机制、推理量化这些基础细节重新理了一遍
- **尝试多种开源模型和开源框架**：本地跑了多个开源大模型，也体验了不同的 Agent 框架和推理前端

这部分看起来「不直接产出」，但对后续做产品选型、判断某个技术方案能不能落地，价值非常大。

## 为什么一定要写博客：两个理由，对内对外各一个

上面这一大串，如果我今天不坐下来写这篇文章，再过两周我自己也记不全了。这就引出了「为什么要记录」这个话题。

做独立开发和在公司上班最大的区别是什么？

不是技术栈，不是收入，不是压力。是：**没有 deadline、没有周报、没有人替你 track 进度。**

自由是真自由。没有人早上站会催你、没有人周五要你交周报、没有人在 Slack 上 ping 你问「那个功能怎么样了」。

但代价也很明确——时间非常容易「溜走」。钻一个技术细节花了三天，回头看主线任务一点没动，这种事过去一个月发生了不止一次。在公司里，项目经理或者迭代截止日期会把你从牛角尖里拉出来；自己做自己老板时，这个外力不存在。

所以记录的目的有两个，刚好一内一外：

### 1. 给自己看：让时间可追溯，让决策可复盘

每天写一篇短小的日记、每周写一篇总结，回头看的时候就能知道：

- 时间到底花在了哪里？是产品开发、是基础设施、是技术研究，还是不知不觉被好奇心带偏到了某个角落
- 哪些决策是有效的？哪些是弯路？下次遇到类似场景可以直接复用经验，不用重新踩一遍坑
- 几年之后再回头翻，应该也是一件挺有意思的事——相当于自己给自己写了一本「独立开发编年体」

### 2. 对外同步：提前建立连接，收到真实反馈

独立软件的开发状态、技术选型的思考、踩过的坑——这些东西提前写出来有两层好处：

- **收到反馈**：可能有前辈一眼能看出你某个方案会踩坑，也可能有人刚好遇到同样的问题可以互相讨论
- **让感兴趣的人提前了解产品方向**：产品还没正式上线就开始积累关注，比憋到 1.0 才「召开发布会」要自然得多

## 记录计划：日更加周更，不追求篇幅

说起来容易做起来难。为了不让「写博客」这件事本身变成负担，我把计划定得非常克制：

### 日更：短小的流水账，10 分钟写完

每天结束前花 10 分钟，用流水账的形式记下三件事：
1. 今天做了什么（一句话概括核心产出）
2. 遇到了什么问题或踩了什么坑
3. 明天打算做什么（最多三件）

不追求篇幅、不追求文笔、不追求逻辑优美。**追求的是「能坚持」。** 如果日更的门槛太高，几天就放弃了，那还不如不开始。

### 周更：每周一篇总结加系统性笔记

每周日或下周一，用一篇长文做三件事：
1. 梳理本周的产出：做了什么、进度如何、离目标还差多远
2. 复盘遇到的问题：哪些是新坑、哪些是老问题反复出现、下周怎么规避
3. 写一些稍成体系的技术笔记或想法：把日常零散的思考整理成结构化的内容，既是对自己思路的梳理，也对读者更有价值

## 写在最后：从这一篇开始

一个月前的今天，我辞掉最后一份外部合作，正式开始独立开发之路。那时候觉得「最大的挑战是技术和产品」，一个月下来发现：**最大的挑战其实是自己——如何在没有任何外部约束的时候，依然能把时间和精力稳稳地投在真正重要的事情上。**

写博客这件事，本质上就是给自己装一个「进度仪表盘」。你没法再骗自己「我今天挺忙的」，因为昨天的日志里写的是「解决了 A 问题」，今天的日志里如果还是「还在解决 A 问题」，那就是进度没动。

这件事我打算长期做下去。

如果你也是独立开发者、或者正在考虑走这条路，欢迎关注这个系列。如果你有自己的「对抗自由陷阱」的好办法，也欢迎在评论区留言，我一个个认真看。
